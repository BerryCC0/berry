# Berry OS

Browser-based operating system for Nouns DAO. Mac OS 8 aesthetic, Ponder indexer, multichain wallet support.

## Two Services, One Database

Berry OS is two deployable services sharing a single Neon Postgres database:

1. **Next.js frontend** (`/`) вЂ” Vercel. The OS shell, apps, API routes, and user-facing everything.
2. **Ponder indexer** (`/ponder`) вЂ” Railway. Indexes 11 Nouns smart contracts into Postgres.

Both auto-deploy on push to main. Both use the same `DATABASE_URL`.

The database contains:
- **Ponder tables** (`ponder_live.*`) вЂ” indexed on-chain data (nouns, auctions, proposals, votes, clients, treasury)
- **App tables** вЂ” user persistence (profiles, settings, themes, window states), KYC verifications, proposal drafts, short links

## Data Flow

**On-chain в†’ user:**
Contract event в†’ Ponder handler indexes to DB в†’ `ponder_live.*` view exposes it в†’ Next.js API route queries via SQL в†’ React hook delivers to component

**User в†’ on-chain:**
Component в†’ wagmi hook в†’ wallet signs в†’ contract call (with BERRY_CLIENT_ID=11) в†’ event emitted в†’ Ponder re-indexes

**Key rule:** The frontend queries `ponder_live.*` views directly via SQL (`app/lib/ponder-db.ts`). All 25+ API routes use `ponderSql()`. The Ponder indexer has no REST API вЂ” it only writes to the database.

## File Authority

| Files | Source | Safe to edit? |
|-------|--------|--------------|
| `app/lib/nouns/abis/*.ts` | Copied from Etherscan verified contracts | Yes, but must match on-chain ABI exactly |
| `app/lib/nouns/utils/image-data.ts` | Static Nouns trait bitmap data (64KB) | No вЂ” immutable reference data |

| `ponder/ponder.schema.ts` | Hand-written, drives DB table creation | Yes, but changes require indexer redeployment |
| `ponder/src/**` | Hand-written indexer handlers | Yes |
| `src/OS/**` | Hand-written OS layer | Yes |
| `app/api/**` | Hand-written API routes | Yes |
| `messages/*.json` | Hand-written i18n strings | Yes |
| `docs/` | Gitignored working docs, not versioned | Yes (but won't be committed) |

**Ponder imports ABIs from the frontend:** `ponder/ponder.config.ts` imports from `../app/lib/nouns/abis/`. The Dockerfile copies these at build time. If you update an ABI, both services get the change.

## Sync Points

These pairs must stay in agreement. Changing one without the other causes silent breakage:

- **Ponder schema в†” Frontend SQL queries.** If you add/rename a column in `ponder.schema.ts`, update every `ponder_live.*` query that references it.
- **ABI files в†” Ponder handlers.** If an ABI adds a new event, add a handler in `ponder/src/` to index it.
- **`OSAppConfig.ts` в†” `[appId]/[[...params]]/page.tsx`.** App IDs in the config must match the deep link route parser.
- **`CLIENT_NAMES` в†” on-chain client registry.** `src/OS/lib/clientNames.ts` maps client IDs to display names. New clients need manual addition.
- **`TREASURY_TOKENS` в†” actual treasury holdings.** `app/lib/nouns/treasury.ts` defines which ERC-20s to track. New treasury tokens need manual addition.
- **i18n message keys в†” component usage.** Components reference keys from `messages/en.json`. Renamed keys break translations.

## Invariants

Rules that must always hold. Violating these causes subtle bugs, not crashes:

- **Client ID 11 on every bid and vote.** `BERRY_CLIENT_ID` must be passed to `createBid` and `castRefundableVoteWithReason`. Without it, Berry OS doesn't earn protocol rewards but the transaction still succeeds.
- **Query `ponder_live.*`, never raw schema.** Raw Ponder schemas are per-deployment (`ponder_abc123.*`). Using them directly works until the next indexer redeployment, then silently returns stale data.
- **CSS Modules only.** No inline styles, no Tailwind, no CSS-in-JS. All styling through `.module.css` files with `--berry-*` custom properties.
- **Zustand for global state.** No React Context for shared state. Context is only used for providers (theme, platform, i18n, Web3).
- **Platform-specific CSS split.** Components with different desktop/mobile layouts use `Component.desktop.module.css` and `Component.mobile.module.css`, selected at runtime via `usePlatform()`.
- **No `any` types.** Use `unknown` with type guards.
- **Singleton enforcement.** Apps marked `singleton: true` in OSAppConfig only allow one window instance. The AppLauncher enforces this.
- **Error boundaries around every app.** WindowManager wraps each app in `AppErrorBoundary`. Never remove this вЂ” it prevents app crashes from killing the OS.

## How to Add Things

### New App

1. Create directory in `src/OS/Apps/YourApp/`
2. Create `YourApp.tsx` implementing `AppComponentProps` (receives `windowId`, `onClose`, `onTitleChange`, `onStateChange`, `initialState`)
3. Create `YourApp.module.css` (and `.desktop.module.css` / `.mobile.module.css` if platform-specific)
4. Create `index.ts` barrel export
5. Add entry to `OSAppConfig.ts` with id, name, category, window config, permissions, component
6. Add icon to `public/icons/`
7. If deep-linkable: add route handling in `app/[appId]/[[...params]]/page.tsx`

### New API Route

1. Create `app/api/your-route/route.ts`
2. For on-chain data: import `ponderSql` from `@/app/lib/ponder-db` and query `ponder_live.*`
3. For user data: import `neon` from `@neondatabase/serverless` and query app tables directly
4. Always validate inputs, use parameterized queries, return proper status codes

### New Indexed Event

1. Add or update the contract ABI in `app/lib/nouns/abis/`
2. Add the event handler in the appropriate `ponder/src/` directory
3. If it creates a new table: add it to `ponder/ponder.schema.ts`
4. Redeploy the indexer (push to main)
5. Add frontend queries after the indexer has synced

### New Language

1. Add locale code to `src/OS/lib/i18n/config.ts` locales array
2. Create `messages/{locale}.json` with all keys from `messages/en.json`
3. Add import in `src/OS/lib/i18n/TranslationProvider.tsx`

## What Breaks Quietly

These failures don't throw errors but cause incorrect behavior:

- **Missing client ID on bids/votes** вЂ” Transaction succeeds but Berry OS earns no rewards
- **Querying raw ponder schema** вЂ” Works until next indexer redeployment, then returns stale data
- **Missing ENS resolution** вЂ” Shows raw 0x addresses instead of names. Not an error, just ugly.
- **Stale CLIENT_NAMES** вЂ” New on-chain clients show as "Client #N" instead of their name
- **Stale TREASURY_TOKENS** вЂ” New treasury holdings invisible in the Treasury app
- **Missing i18n key** вЂ” Shows the key path string instead of translated text

## Architecture Patterns

**State management:** Zustand stores emit events via systemBus for cross-system communication. Stores are the source of truth; the event bus is for reactions.

**Persistence:** Without wallet в†’ InMemoryAdapter (ephemeral). With wallet в†’ NeonAdapter (persistent). PersistenceManager handles upgrade/downgrade automatically. Auto-save is debounced at 2 seconds.

**Three-tier event bus:** SystemBus (OS internals), AppBus (app-to-app), BridgeBus (OSв†’App read-only). Prevents circular dependencies.

**Platform detection:** `usePlatform()` returns desktop/mobile/tablet/farcaster. Components select CSS module at runtime. Windows are fullscreen on mobile, draggable on desktop.

**Lazy loading:** All apps are dynamically imported via React.lazy in WindowManager. Each app is wrapped in Suspense + ErrorBoundary.

**ENS resolution:** Three-tier cache: in-memory (24h, 10k entries) в†’ `ponder_live.ens_names` table в†’ HTTP fallback to ensideas.com.

## Conventions

- **File naming:** PascalCase for components (`Window.tsx`), camelCase for utilities (`clientNames.ts`)
- **Path aliases:** `@/*` (root), `@/OS/*` (`src/OS/*`), `@/Apps/*` (`src/Apps/*`)
- **CSS variables:** All theme values use `--berry-*` prefix
- **Ponder queries:** Always use `ponder_live.` prefix, never raw schema names
- **Ponder plurals:** List queries use double-plural (`nounss`, `proposalss`) вЂ” this is a Ponder convention
- **Barrel exports:** Each module directory has an `index.ts` re-exporting its public API
